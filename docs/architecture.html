ภาพรวม & หลักการSystem overview & design principles

bogbogprox นั่งคั่นกลางระหว่าง browser กับ target — decrypt HTTPS แล้วเปิดทางให้คุณ เห็น จับ แก้ และสแกน traffic ทุกอย่าง. โมเดลเดียวกับ Burp classic แต่เขียนใหม่ทั้งกองด้วย Rust.

Data plane vs Control planeสองระนาบที่แชร์ coordinator ชุดเดียว

แยกชัดสองระนาบ แต่ทั้งคู่ถือ coordinator ตัวเดียวกัน ผ่าน Arc — นี่คือกลไกที่ทำให้ "กดปุ่มบน dashboard" ไปมีผลกับ "traffic ที่กำลังวิ่ง" ได้ทันที.

  • Data plane (bogbogprox-engine) — เส้นที่ traffic จริงวิ่ง. ต้องเร็ว, ต่อ connection. hudsucker เรียก handler ต่อ request/response.
  • Control plane (bogbogproxd REST/SSE) — คน/AI สั่งงาน: toggle intercept, add rule, ยิง intruder.

วงจรชีวิต request / responseลำดับ hook — สำคัญมาก

หัวใจอยู่ที่ handle_request / handle_response. แต่ละชั้นทำ ก่อนชั้นถัดไป — ลำดับนี้คือ contract ของทั้งระบบ:

ขาไป — request

  1. CONNECT? → ปล่อยผ่าน ตั้ง TLS tunnel
  2. Upgrade: websocket? → ปล่อยผ่านไป WS handler
  3. buffer body ≤ 16 MiB — เกินตั้ง body_truncated
  4. Match & Replace apply_request + expand {{var}}
  5. Plugins on_request ข้ามถ้า body truncated
  6. Intercept ถ้า on & in-scope → await การตัดสินใจของคน
  7. rebuild_parts ถ้า dirty → สร้าง wire ใหม่
  8. store.insert_request เก็บ "ที่ส่งจริง" + broadcast FlowNew → forward

ขากลับ — response

  1. buffer body ≤ 16 MiB
  2. Match & Replace apply_response
  3. Plugins on_response
  4. Intercept response ถ้า on
  5. store.attach_response + broadcast FlowUpdate
  6. Scanner.scan → broadcast Finding → ตอบ browser

🌐 Browser integration เชิงลึกproxy protocol · CONNECT · TLS MITM · CA · managed browser

ตั้งต้นให้ตรง bogbogprox ไม่มี browser ของตัวเอง — มันคือ HTTP proxy ที่ browser (ของคุณ) ชี้มาหา เหมือน Burp classic. ส่วนนี้อธิบายตั้งแต่ TCP จนถึง TLS MITM.

5.1 · browser เชื่อมยังไง — proxy protocol

ตั้ง browser ให้ใช้ HTTP proxy 127.0.0.1:8888 แล้ว:

  • HTTP ธรรมดา — browser ส่งแบบ absolute-form: GET http://site/path ให้ proxy อ่าน host จาก URI แล้ว forward
  • HTTPS — browser ส่ง CONNECT site:443 ขอ tunnel ก่อน. ปกติ proxy จะเปิดท่อเฉยๆ ไม่เห็นเนื้อ — แต่เราจะ MITM

5.2 · TLS MITM ทำยังไง (hudsucker)

  1. รับ CONNECT → ตอบ 200 เริ่ม TLS handshake กับ browser โดยสวมเป็น site
  2. ออก leaf cert ของ site สดๆ (rcgen) ลงนามด้วย BogBogProx CA แล้ว cache ไว้
  3. browser handshake สำเร็จ เพราะ trust BogBogProx CA → คุย plaintext กับ proxy
  4. proxy อ่าน request ที่ decrypt แล้ว มาถึง handler เป็น origin-form · เปิด TLS เส้นสองไป site จริง · forward

ผลคือ proxy เห็นและแก้ HTTPS ได้เต็ม ทั้งที่ browser กับ site ยังคิดว่าคุย TLS ปกติ.

5.3 · CA & trust — ทำไมต้องติดตั้ง cert

BogBogProx สร้าง CA เฉพาะเครื่อง (bogbogproxd ca generate → ~/.config/bogbogprox/ca/bogbogprox-ca.pem + .key perms 0600). leaf cert ทุกใบลงนามด้วย CA นี้; browser จะ trust ก็ต่อเมื่อ ติดตั้ง bogbogprox-ca.pem ใน trust store.

ระวัง ห้ามแชร์/commit CA นี้ — ใครมี key = MITM traffic ของคุณได้. .gitignore กัน *.key/*.pem ไว้แล้ว. ถ้าไม่ติดตั้ง → browser ขึ้น cert error.

เพื่อความสะดวก (เหมือน Burp embedded browser) มี subcommand เปิด Chromium ที่ตั้งค่าให้เสร็จ:

5.5 · WebSocket

Upgrade: websocket ถูกปล่อยผ่าน ไม่ buffer (ไม่งั้นรบกวน upgrade); hudsucker dispatch ไป WsHandler แยก — แต่ละ message log ที่ WsLog แล้ว broadcast WsMessage. wss:// MITM ได้ผ่าน CONNECT+TLS เหมือน HTTPS.

5.6 · ทำไม HTTP/1.1 อย่างเดียว

build hudsucker ไม่เปิด feature http2 โดยตั้งใจ → ทั้ง browser (ALPN) และ upstream เป็น h1. ผลดี: request บน connection เดียว serialize ไม่ multiplex → จับคู่ req↔resp แบบ FIFO ถูกต้อง 100% โดยไม่ต้องมี stream-id correlation. แลกกับบางไซต์ที่ชอบ h2 อาจ behave ต่างเล็กน้อย (ทุกไซต์รองรับ h1 fallback).

Storage architectureFlowStore port · SQLite mutex · Postgres DB-actor

Port: trait FlowStore — insert_request / attach_response / list_flows / get_flow / count / clear. method เป็น sync (เรียกจาก async engine แบบ blocking — ยอมรับได้).

SQLite · local Arc<Mutex<Connection>> (single-writer), WAL mode, headers เป็น JSON, body เป็น BLOB, perms 0600.
Postgres · team sync client มี tokio runtime ของตัวเอง → เรียกใน tokio worker = panic. แก้ด้วย DB-actor thread.
